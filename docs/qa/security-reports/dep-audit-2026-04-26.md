### Finding MOD-001 — PostCSS XSS via CSS Stringify

| Campo | Valor |
|-------|-------|
| **Pacote** | `postcss` < 8.5.10 (bundled interno do Next.js) |
| **Severidade** | MODERATE (CVSS 6.1) |
| **CVE** | GHSA-qx2v-qp2m-jg93 |
| **CWE** | CWE-79 (XSS via `</style>` não escapado no output CSS) |
| **Via** | `next` → `next/node_modules/postcss` |
| **Fix sugerido pelo npm** | Downgrade `next@9.3.3` ← **INCORRETO / NÃO APLICAR** |

**Análise de risco real:**

> O npm audit sugere downgrade para `next@9.3.3` mas o projeto usa **Next.js 16.2.4**. Esta sugestão é um artefato do registro npm — aplicá-la destruiria o projeto. O finding aponta para o `postcss` **bundled internamente** pelo Next.js para processamento de CSS em **build-time**, não em runtime com input do usuário.
>
> **Exploitabilidade real: BAIXA** — a vulnerabilidade XSS exige que CSS não-confiável de usuário seja processado via `postcss.stringify()` em runtime. O AutoPost não faz isso — o PostCSS é usado pelo Next.js apenas durante o build para processar Tailwind CSS. Não há vetor de ataque.

**Decisão:** Documentar como dívida técnica LOW. Monitorar releases do Next.js — quando atualizarem o postcss bundled para ≥8.5.10 a vulnerabilidade some automaticamente.
