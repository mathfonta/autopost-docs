## Story
**As a** profissional em campo que acabou de gravar um vídeo de obra,
**I want** enviar o vídeo direto para o AutoPost sem precisar comprimir manualmente antes,
**so that** eu não perca tempo no meio do serviço e o post seja criado sem fricção.

## Contexto / Motivação
Vídeo real reportado: `VID_20260508_155315.mp4` — 115 MB, 1080×1920, 45s, MP4.
Falhou silenciosamente duas vezes porque `MAX_FILE_SIZE = 20 MB` no backend rejeitou o upload.
O cliente não recebeu mensagem clara — apenas um erro genérico que sumiu rápido.

O cliente não pode (e não deve) comprimir o vídeo manualmente antes de enviar.
O AutoPost deve absorver essa fricção: receber o vídeo bruto, comprimir internamente, publicar.

## Acceptance Criteria

### Backend
1. **AC1 — Limite elevado para vídeo:** `MAX_FILE_SIZE` para `content_type in ("reels", "story")` é 500 MB. Para fotos, permanece 20 MB.
2. **AC2 — Compressão automática:** Após receber o arquivo de vídeo, antes de salvar no R2, o backend executa `_compress_video()` com FFmpeg. Target: ≤ 50 MB, H.264 + AAC, CRF 28, resolução máxima 1080×1920 (escala proporcional se maior).
3. **AC3 — Fallback gracioso:** Se a compressão falhar (FFmpeg não disponível, timeout), o vídeo original é salvo sem interrupção — desde que ≤ 200 MB.
4. **AC4 — Railway proxy:** O limite do proxy Railway é configurado para aceitar até 500 MB via variável de ambiente ou `railway.toml`.

### Frontend — Feedback de tamanho
5. **AC5 — Badge informativo na seleção:** Quando o usuário seleciona um vídeo > 20 MB, o `UploadScreen` exibe um badge não-bloqueante: `"📹 Vídeo grande detectado (Xmb) — comprimindo automaticamente"`.
6. **AC6 — Etapa "Comprimindo vídeo" no GeneratingScreen:** Para `content_type in ("reels", "story")`, a `GeneratingScreen` exibe uma etapa adicional entre "Foto enviada" e "Analisando conteúdo": `"Comprimindo vídeo..."` — marcada como concluída assim que o upload é confirmado pelo backend.

### Frontend — Dicas de gravação
7. **AC7 — Card de dicas no UploadScreen:** Para `content_type in ("reels", "story")`, o `UploadScreen` exibe um card colapsável com título "📱 Dicas para gravar vídeos melhores" antes do botão de seleção de arquivo.
8. **AC8 — Conteúdo genérico (sem menção a marca/OS):** O card contém exatamente:
   - **Configuração da câmera:** Grave em 1080p — o Instagram não usa resolução acima disso. Evite 4K. Verifique nas configurações da câmera do seu celular.
   - **Na hora de gravar:** 15–45 segundos é o ideal. Boa iluminação natural = arquivo menor e vídeo mais bonito. Segure firme — vídeo tremido aumenta o tamanho do arquivo.
   - **Se ficou grande mesmo assim:** O AutoPost comprime automaticamente — é só enviar normalmente.
9. **AC9 — Card colapsado por padrão:** O card começa recolhido para não poluir o fluxo. Usuário expande se quiser.

### Qualidade
10. **AC10 — Sem regressão em fotos:** O fluxo de upload de fotos (feed_photo, carousel, before_after) não é afetado. `MAX_FILE_SIZE` de 20 MB permanece para fotos.
11. **AC11 — Testes:** `test_video_compression.py` com pelo menos 3 casos: vídeo pequeno (< 20 MB, sem compressão), vídeo grande (> 50 MB, comprimir), FFmpeg indisponível (fallback).

## Scope

**IN:**
- `app/api/content.py` — limite diferenciado por tipo (vídeo 500 MB / foto 20 MB) + chamar `_compress_video()` antes de salvar no R2
- `app/core/video.py` (novo) — `_compress_video(video_bytes: bytes) -> bytes` com FFmpeg subprocess, CRF 28, H.264 + AAC, fallback gracioso
- `railway.toml` ou variável de ambiente — aumentar limite do proxy para 500 MB
- `components/dashboard/GeneratingScreen.tsx` — etapa "Comprimindo vídeo..." para reels/story
- `components/dashboard/UploadScreen.tsx` — badge de tamanho + card colapsável de dicas
- `tests/test_video_compression.py` — 3+ casos de teste

**OUT:**
- Compressão de fotos (já tem `_compress_image_for_claude()`)
- Editor de texto overlay no vídeo (story futura)
- Preview do vídeo comprimido antes do upload
- Qualquer mudança em `publish_post` ou no pipeline além do upload

## Business Value
Desbloqueia o uso real de Reels por qualquer cliente com celular moderno.
Vídeos de câmera nativa chegam a 100–200 MB. Sem este fix, o recurso de Reels é inutilizável para 90% dos clientes sem compressão manual prévia.

## Dev Notes
- AC4 (Railway proxy): limite de body size não é configurável via railway.toml. O backend aceita 500MB, mas vídeos > ~100MB podem ser rejeitados pelo proxy Railway antes de chegar à app. Documentado como risco em railway.toml. Solução definitiva: upload direto ao R2 via presigned URL (backlog).
- AC6 timing (conforme observação @po): a compressão acontece sincronamente no POST /content-requests, então quando GeneratingScreen abre o upload já está concluído — etapa "Comprimindo vídeo" corretamente marcada como `state="done"`.
- `npm run lint` não existe neste projeto frontend — validado com `npm run typecheck` (tsc --noEmit).
- 7 testes criados (AC11 pede 3+): pequeno, boundary, grande+comprimido, FFmpeg ausente, returncode!=0, timeout, quicktime.
