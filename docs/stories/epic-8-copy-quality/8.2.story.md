# Story 8.2 — Tom de Voz por Perfil

**Epic:** 8 — Qualidade e Personalização da Copy
**Status:** Done
**Agente executor:** @dev
**Data criação:** 2026-04-27

---

## Descrição

Atualmente o Agente Copywriter gera copy com tom neutro/genérico, independente do perfil da empresa. Para negócios de construção e reforma (como a Espectra), o tom ideal pode variar: uma construtora de alto padrão prefere tom formal e aspiracional, uma empresa de reformas rápidas prefere tom descontraído e direto.

Esta story adiciona a configuração de **tom de voz** ao perfil do usuário. O usuário define uma vez nas configurações e o Copywriter usa esse tom em todas as gerações subsequentes.

---

## Acceptance Criteria

- [ ] AC1: Campo `voice_tone` (enum: `formal` | `casual` | `technical`) existe na tabela de perfil do usuário no banco
- [ ] AC2: Migration criada e aplicada sem breaking changes — default `casual` para usuários existentes
- [ ] AC3: Endpoint `PATCH /auth/profile` aceita e persiste `voice_tone`
- [ ] AC4: Agente Copywriter recebe `voice_tone` e injeta instrução de tom no prompt (`#TOM_DE_VOZ`)
- [ ] AC5: Interface de configuração de tom disponível no Dashboard (seletor visual com 3 opções)
- [ ] AC6: Tom selecionado persiste entre sessões e é carregado ao abrir o app
- [ ] AC7: Pipelines sem `voice_tone` definido usam `casual` como fallback — backward compatible

---

## Escopo

**IN:**
- Campo `voice_tone` no modelo de usuário (backend + migration)
- Endpoint PATCH para atualizar perfil
- Injeção do tom no prompt do Copywriter
- Seletor de tom no frontend (Dashboard ou página de configurações)
- Carregamento do tom no contexto do pipeline Celery

**OUT:**
- Tom por post individual (isso seria outra story — o tom é configuração global do perfil)
- Histórico de mudanças de tom
- Tom customizado (texto livre) — apenas as 3 opções predefinidas
- Aplicação retroativa em posts já gerados

## Dev Notes

### Contexto da Story 8.1 (aprendizados aplicáveis)
- O padrão de passar dados extras pelo pipeline foi estabelecido com `user_context`
- A mesma abordagem se aplica: campo no model → API → pipeline task → agente
- O Copywriter já tem ponto de injeção de contexto (`#CONTEXTO_DO_USUARIO`) — criar `#TOM_DE_VOZ` no mesmo padrão

### Backend — Modelo e Migration

**Tabela:** `content_requests` NÃO — o tom vai na tabela de usuários/perfil da empresa.

Verificar onde fica o perfil do usuário. Provavelmente `users` ou tabela de perfil da empresa no Supabase. Checar `app/models/` para identificar o modelo correto.

**Migration:**
```sql
ALTER TABLE users ADD COLUMN IF NOT EXISTS voice_tone VARCHAR(20) DEFAULT 'casual'
  CHECK (voice_tone IN ('formal', 'casual', 'technical'));
```

### Backend — Endpoint

Verificar se existe `PATCH /auth/profile` ou `PUT /auth/me`. Se não existir, criar.

**Request body (adicionar ao schema existente):**
```python
voice_tone: Optional[Literal['formal', 'casual', 'technical']] = None
```

### Backend — Pipeline

Em `app/tasks/pipeline.py`, no momento em que o pipeline busca os dados do usuário, adicionar o `voice_tone` ao contexto passado para o Copywriter (mesmo padrão do `user_context`).

### Backend — Copywriter Agent

Em `app/agents/copywriter.py`, adicionar no prompt:

```python
TOM_MAP = {
    'formal': 'tom formal, profissional e aspiracional',
    'casual': 'tom descontraído, próximo e direto',
    'technical': 'tom técnico, preciso e informativo',
}
# Injetar como: #TOM_DE_VOZ: {TOM_MAP.get(voice_tone, TOM_MAP['casual'])}
```

### Frontend — Seletor de Tom

Adicionar seletor no Dashboard (próximo ao header ou em uma seção de configurações inline). Três botões/cards:

| Opção | Label | Descrição curta |
|-------|-------|-----------------|
| `formal` | Formal | Profissional e aspiracional |
| `casual` | Descontraído | Próximo e direto |
| `technical` | Técnico | Preciso e informativo |

Persistir via `PATCH /auth/profile` ao selecionar. Carregar o tom atual via `GET /auth/me`.
