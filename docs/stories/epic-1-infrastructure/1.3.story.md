# Story 1.3 — Multi-Tenancy + Row Level Security

**Status:** Done

## Descrição

Como plataforma SaaS multi-tenant, preciso garantir que cada cliente acesse apenas seus próprios dados, tanto via aplicação (application-level isolation) quanto via banco de dados (Row Level Security no PostgreSQL).

---

## Critérios de Aceitação

- [x] AC1: `TenantMixin` disponível para todos os modelos multi-tenant — adiciona `client_id` FK com CASCADE e índice
- [x] AC2: `require_ownership(resource_client_id, current_client)` levanta 403 quando IDs diferem
- [x] AC3: `tenant_filter(query, model, current_client)` adiciona `WHERE client_id = ?` automaticamente
- [x] AC4: RLS habilitado na tabela `clients` com `FORCE ROW LEVEL SECURITY`
- [x] AC5: Política `clients_select_own` — usuário vê apenas o próprio registro (`auth.uid() = id`)
- [x] AC6: Política `clients_update_own` — usuário atualiza apenas o próprio registro
- [x] AC7: INSERT e DELETE na tabela `clients` bloqueados para user/anon key (service_role only)

## Dev Notes

- `declared_attr` para `__table_args__` no mixin causa recursão infinita — usar `index=True` na coluna diretamente
- PostgreSQL compila UUIDs sem hífens em literals SQL (`str(uuid).replace("-", "")` nos testes)
- service_role key bypassa RLS — backend sempre tem acesso total
- FORCE ROW LEVEL SECURITY garante que mesmo o owner do schema seja afetado pelas políticas
