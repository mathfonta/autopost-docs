## Story

**Como** responsável pelo produto AutoPost,
**eu quero** uma tarefa periódica que mantenha o projeto Supabase (plano free) ativo,
**para que** o banco nunca auto-pause por inatividade e derrube o app inteiro sem aviso.

## Contexto técnico

Projetos Supabase no plano free auto-pausam após ~7 dias sem atividade de query. Quando isso acontece, o pooler (Supavisor) passa a responder com `ENOTFOUND tenant/user postgres.<ref> not found` e o DNS de `db.<ref>.supabase.co` deixa de resolver — o endpoint `/health` (`backend/app/api/health.py:19`) já expõe isso como `database: "error: ..."` e `status: "degraded"`. A correção do incidente de 2026-07-22 foi manual (restore no dashboard Supabase). Esta story elimina a recorrência sem custo adicional, usando a infraestrutura Celery Beat que já roda em produção (serviço `beat` no `railway.toml`, ver `app/tasks/__init__.py:21-27` — já existe uma entrada agendada, `weekly-content-intelligence`).

## Acceptance Criteria

1. Nova task Celery `pipeline.keepalive_ping` executa uma query trivial (`SELECT 1`) contra o banco via `WorkerSessionLocal` (padrão já usado por outras tasks do worker, `app/core/database.py:34-38`)
2. Task registrada em `celery_app.conf.beat_schedule` (`app/tasks/__init__.py`) com frequência de **a cada 3 dias** (margem de segurança confortável abaixo do limite de ~7 dias de inatividade do free tier)
3. Falha na query (ex.: banco já pausado, rede instável) é capturada e logada — **nunca** propaga exceção não tratada nem derruba o worker/beat
4. Task não depende de nenhum modelo de negócio (`ContentRequest`, etc.) — só abre sessão e roda `SELECT 1`, para minimizar superfície de falha
5. Suite de testes existente (`pytest backend/tests/`) continua passando sem alteração
6. Teste novo (`backend/tests/test_keepalive.py` ou incluído em arquivo existente de tasks) cobre: (a) execução bem-sucedida, (b) captura de exceção sem propagar
7. Resultado documentado no Segundo Cérebro (`*cerebro-update`) — aprendizado do incidente + solução aplicada

## Dev Notes

### Padrão de task Celery síncrona sobre coroutine async

Todas as tasks em `pipeline.py` seguem o mesmo padrão: função `async def` interna + `_run_sync(coro)` (definida em `pipeline.py:31`) para rodar dentro do worker Celery (que é síncrono). Replicar esse padrão exatamente — não introduzir um novo estilo de execução.

[Source: backend/app/tasks/pipeline.py:31, 187-284]

### Sessão de banco no contexto de worker

Usar `WorkerSessionLocal` (não `AsyncSessionLocal`, que é para o contexto FastAPI/request) — `WorkerSessionLocal` usa `NullPool` porque cada `asyncio.run()` do Celery cria seu próprio event loop, e um pool persistente entre loops causa erro.

[Source: backend/app/core/database.py:27-38]

### Beat Schedule existente — reaproveitar, não recriar infraestrutura

`app/tasks/__init__.py:21-27` já tem `celery_app.conf.beat_schedule` com uma entrada (`weekly-content-intelligence`). O serviço `beat` do Celery já roda em produção no Railway (`railway.toml`, serviço `beat`). Esta story só adiciona uma segunda entrada ao dicionário — nenhuma infraestrutura nova, nenhum novo serviço Railway, custo zero.

[Source: backend/app/tasks/__init__.py:1-44]
[Source: railway.toml — serviço "beat"]

### Health check como referência de diagnóstico

`backend/app/api/health.py:19-52` é o endpoint que expôs o incidente (`database: "error: ..."`). Não precisa ser alterado nesta story — só serve de referência para confirmar que o keep-alive está funcionando (rodar `curl .../health` periodicamente continua sendo a forma de verificar o estado real, não substitui o keep-alive).

[Source: backend/app/api/health.py]
