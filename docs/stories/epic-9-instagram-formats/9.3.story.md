## Descrição

Com as Stories 9.1 e 9.2, o usuário já escolhe Formato + Sub-estratégia e o Copywriter gera copy calibrada. Porém, o upload ainda é de **imagem única** para todos os formatos — o UploadScreen não diferencia Carrossel (múltiplas fotos), Reels (vídeo) ou Story (imagem ou vídeo).

Esta story torna o `UploadScreen` **consciente do formato selecionado**:

- **Feed Photo** → comportamento atual, sem mudança
- **Carrossel** → seleção de 2 a 10 imagens com grid de preview e contador
- **Reels** → upload de 1 vídeo (.mp4 / .mov) com indicador de tipo
- **Story** → upload de 1 arquivo (imagem ou vídeo, auto-detect por MIME)

O backend já suporta `photos[]` e valida contagem para Carrossel. O principal gap é o frontend (UploadScreen format-aware + multi-select) e a aceitação de vídeo no backend (MIME types + chave de arquivo .mp4).

## Valor de Negócio

Viabiliza os dois formatos de maior alcance orgânico no Instagram 2025–2026: Carrossel (maior taxa de saves e shares) e Reels (maior distribuição para não-seguidores). Sem esta story, o usuário seleciona esses formatos mas sobe uma imagem avulsa — gerando copy calibrada para um formato que o upload não suporta. O impacto direto é desbloquear a proposta de valor diferencial do produto.

## Decisões Técnicas

- `UploadScreen` recebe prop `postTypeId` (já tem) e deriva a UI da lógica interna — sem criar 4 componentes separados
- Multi-select no Carrossel: `<input multiple accept="image/*">` + estado `pendingFiles: File[]`
- Grid de preview: thumbnails 80×80 com badge "✕" para remover, sem reordenação
- Reels/Story: `<input accept="video/mp4,video/quicktime,image/*">` para Story; `accept="video/mp4,video/quicktime"` para Reels
- Backend: `MULTI_PHOTO_TYPES` adiciona `"reels"` e `"story"` como tipos que aceitam vídeo (mas não como multi-photo)
- Extensão: detectar por MIME — se `video/*` → `.mp4`, se `image/*` → `.jpg`
- Validação Reels: `n == 1` e MIME video; validação Story: `n == 1` e MIME image ou video
- `dashboard/page.tsx`: `pendingFiles` para Carrossel, `pendingFile` continua para os demais — sem quebrar o fluxo de Preview que espera URL único

## Dev Notes

### UploadScreen — Lógica de variação

```tsx
// Derivar modo baseado no postTypeId
const mode = postTypeId === "carousel"
  ? "multi-image"
  : postTypeId === "reels"
    ? "video"
    : postTypeId === "story"
      ? "image-or-video"
      : "single-image"; // feed_photo
```

### Carrossel — multi-select

```tsx
// Input hidden
<input
  ref={galleryRef}
  type="file"
  accept="image/*"
  multiple
  onChange={handleMultiFiles}
  className="hidden"
/>

// Handler acumula arquivos
function handleMultiFiles(e: ChangeEvent<HTMLInputElement>) {
  const newFiles = Array.from(e.target.files ?? []);
  setPendingFiles(prev => [...prev, ...newFiles].slice(0, 10));
}
```

### FormData — envio

```ts
// Carrossel: photos[]
if (postTypeId === "carousel") {
  pendingFiles.forEach(f => formData.append("photos", f));
} else {
  formData.append("photo", pendingFile);
}
```

### Backend — ALLOWED_CONTENT_TYPES

```python
ALLOWED_IMAGE_TYPES = {"image/jpeg", "image/png", "image/webp"}
ALLOWED_VIDEO_TYPES = {"video/mp4", "video/quicktime"}
ALLOWED_CONTENT_TYPES = ALLOWED_IMAGE_TYPES | ALLOWED_VIDEO_TYPES

# Extensão dinâmica na chave R2
ext = ".mp4" if upload.content_type in ALLOWED_VIDEO_TYPES else ".jpg"
key = f"uploads/{current_client.id}/{uuid.uuid4()}{ext}"
```

### Backend — Validação por formato

```python
# Reels: 1 vídeo
if content_type == "reels":
    if n != 1 or effective_photos[0].content_type not in ALLOWED_VIDEO_TYPES:
        raise HTTPException(422, "reels requer 1 arquivo de vídeo (.mp4 ou .mov).")

# Story: 1 imagem ou vídeo
if content_type == "story":
    if n != 1:
        raise HTTPException(422, "story requer exatamente 1 arquivo.")
    # MIME image ou video — já aceito por ALLOWED_CONTENT_TYPES
```
