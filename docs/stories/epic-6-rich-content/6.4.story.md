# Story 6.4 — Pipeline Multi-Foto: Análise Enriquecida, Design e Publicação

**Status:** Done
**Epic:** 6 — Rich Content
**Story:** 6.4
**Estimativa:** 5 pontos
**Depende de:** Story 6.1 (upload múltiplo)

---

## Descrição

Como sistema AutoPost, quero que o pipeline de geração de conteúdo suporte múltiplas fotos de ponta a ponta — análise individual por foto, geração de copy multi-foto com variação real no retry, processamento de design para formatos before/after e carousel, e publicação dos resultados — para que clientes com múltiplas fotos recebam conteúdo de alta qualidade sem intervenção manual.

---

## Critérios de Aceitação

- [x] AC1: `analyst.py` analisa cada foto individualmente em uploads multi-foto e retorna array `photos` com análise por foto
- [x] AC2: Análise retorna campos enriquecidos: `description` (4-5 frases), `elementos_visuais`, `ambiente`, `nivel_acabamento`
- [x] AC3: `copywriter.py` detecta multi-foto via campo `photos` e constrói descrição agregada para geração de copy
- [x] AC4: Retry de copy usa `_RETRY_APPROACHES` para forçar variação real de abordagem (resultado / emocional / técnico) por tentativa
- [x] AC5: `designer.py` processa before/after com `process_before_after_two_photos()` para content_type `before_after`
- [x] AC6: `designer.py` processa carousel (N fotos, 1 slide por foto) para content_type `carousel`
- [x] AC7: `publisher.py` publica resultados multi-foto
- [x] AC8: `pipeline.py` orquestra os fluxos: foto única (comportamento anterior mantido), before_after, carousel
- [x] AC9: `retry_count` do request é passado como `retry_attempt` ao copywriter para abordagem contextual
- [x] AC10: Testes cobrindo todos os novos fluxos em `test_pipeline.py`

## Dev Notes

- `photo_keys` / `photo_urls` são arrays paralelos — mantido backward-compat com `photo_key` / `photo_url` singulares
- Pipeline usa `content_type == "before_after"` e `content_type == "carousel"` para routing de design
- Análise multi-foto: se qualquer foto for `quality=bad`, o bad prevalece no resultado agregado
- `_RETRY_APPROACHES` é indexado por `retry_attempt` (1, 2, 3+); 3 é o fallback para tentativas além
