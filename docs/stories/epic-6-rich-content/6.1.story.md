# Story 6.1 — Upload Múltiplo + Modelo (Backend)

**Status:** InReview  
**Epic:** 6 — Conteúdo Rico: Múltiplas Fotos, Antes/Depois e Carrossel  
**Story:** 6.1  
**Estimativa:** 3 pontos  
**Depende de:** Story 5.3 (campo `content_type` já no DB e no modelo)

---

## Descrição

Como cliente AutoPost, quero enviar até 10 fotos em um único upload para que o sistema possa compor automaticamente um carrossel ou antes/depois — sem precisar enviar cada foto separadamente ou usar outro aplicativo.

Como desenvolvedor, preciso que `POST /content-requests` aceite múltiplos arquivos e que o modelo `ContentRequest` armazene a lista de chaves e URLs, mantendo retrocompatibilidade total com uploads de foto única existentes.

---

## Critérios de Aceitação

- [ ] **AC1 — Migração backward-compatible:** A migration adiciona `photo_urls JSONB nullable` e `photo_keys JSONB nullable` em `content_requests`, sem alterar ou remover `photo_url` e `photo_key` existentes. Nenhuma linha existente é afetada.
- [ ] **AC2 — Modelo atualizado:** `ContentRequest` tem `photo_urls: Mapped[list | None]` e `photo_keys: Mapped[list | None]` com `JSONB, nullable=True`.
- [ ] **AC3 — Endpoint multi-foto:** `POST /content-requests` aceita campo `photos: list[UploadFile]` (1–10 arquivos) via multipart. O campo legado `photo: UploadFile` continua funcional — se presente, é tratado como `photos` com 1 item.
- [ ] **AC4 — Upload paralelo para R2:** Cada arquivo em `photos` é validado (tipo + tamanho), uploadeado para R2 individualmente. `photo_keys` recebe a lista de chaves; `photo_urls` recebe a lista de URLs presigned frescas.
- [ ] **AC5 — Retrocompat single-photo:** Quando 1 foto é enviada (`photo` ou `photos` com 1 item), `photo_key = photo_keys[0]` e `photo_url = photo_urls[0]` são preenchidos normalmente. O pipeline existente roda sem alteração.
- [ ] **AC6 — Novos `content_type`:** O endpoint aceita `content_type` com valores `before_after` e `carousel` além dos existentes (`post_simples`, `obra_andamento`, `obra_concluida`, `engajamento`, `bastidores`). Valores inválidos retornam 422.
- [ ] **AC7 — Validação de contagem por tipo:** `before_after` exige exatamente 2 fotos (422 caso contrário); `carousel` exige 2–10 fotos (422 caso contrário); tipos simples exigem exatamente 1 foto (422 caso contrário).
- [ ] **AC8 — `_freshen_urls` atualizada:** A função refresh cada URL em `photo_urls` usando as chaves em `photo_keys`. Mantém comportamento existente para `photo_url` singular.
- [ ] **AC9 — Response schema atualizado:** `ContentRequestDetailResponse` inclui `photo_urls: list[str] | None = None`. Respostas de foto única continuam retornando `photo_urls = None` ou `[photo_url]` (decisão do dev — documentar).
- [ ] **AC10 — Regressão zero:** Todos os testes existentes de `POST /content-requests` (foto única) continuam passando. `GET /content-requests` e `GET /content-requests/{id}` não regridem.
- [ ] **AC11 — Cobertura de testes:** Mínimo 12 novos testes cobrindo: multi-foto upload, retrocompat foto única, validação de contagem por `content_type`, 422 para tipos inválidos, 422 para contagem incorreta, `_freshen_urls` com multi-foto.

## Dev Notes

### Estratégia de Retrocompatibilidade

O endpoint atual recebe `photo: UploadFile = File(...)` (obrigatório). A mudança deve ser:

```python
@router.post("", response_model=ContentRequestResponse, status_code=201)
async def submit_photo(
    photo: UploadFile | None = File(None),   # retrocompat — campo legado
    photos: list[UploadFile] | None = File(None),  # novo — multi-foto
    content_type: str | None = Form(None),
    current_client: Client = Depends(get_current_client),
    db: AsyncSession = Depends(get_db),
):
    # Normalizar: unificar em effective_photos
    if photos:
        effective_photos = photos
    elif photo:
        effective_photos = [photo]
    else:
        raise HTTPException(status_code=422, detail="Ao menos uma foto é obrigatória.")
```

### Validação de content_type e Contagem

```python
VALID_CONTENT_TYPES = {
    "post_simples", "obra_andamento", "obra_concluida",
    "engajamento", "bastidores", "before_after", "carousel"
}
MULTI_PHOTO_TYPES = {"before_after", "carousel"}

if content_type and content_type not in VALID_CONTENT_TYPES:
    raise HTTPException(status_code=422, detail=f"content_type inválido: {content_type}")

n = len(effective_photos)
if content_type == "before_after" and n != 2:
    raise HTTPException(status_code=422, detail="before_after requer exatamente 2 fotos.")
if content_type == "carousel" and not (2 <= n <= 10):
    raise HTTPException(status_code=422, detail="carousel requer 2–10 fotos.")
if content_type not in MULTI_PHOTO_TYPES and n != 1:
    raise HTTPException(status_code=422, detail="Tipos simples aceitam apenas 1 foto.")
```

### Upload Multi-foto para R2

keys, urls = [], []
for i, upload in enumerate(effective_photos):

### `_freshen_urls` Atualizada

```python
def _freshen_urls(req: ContentRequest) -> ContentRequest:
    # Retrocompat — foto singular
    try:
        req.photo_url = generate_presigned_url(req.photo_key, PRESIGNED_URL_TTL)
    except Exception:
        pass

    # Multi-foto — regenera lista
    if req.photo_keys:
        fresh = []
        for key in req.photo_keys:
            try:
                fresh.append(generate_presigned_url(key, PRESIGNED_URL_TTL))
            except Exception:
                fresh.append(req.photo_url)  # fallback para foto principal
        req.photo_urls = fresh

    return req
```
