## Story
**As a** cliente no onboarding,
**I want** ver que meu perfil está sendo analisado e receber uma sugestão opcional de ajuste de segmento,
**so that** eu entenda que o sistema está me personalizando e possa aceitar um nicho que me descreva melhor.

## Contexto / Motivação

Fecha o loop do Epic 22 no frontend. Depois que o cliente conecta o Instagram (`connected=true`), o Scout roda em background (Story 22.3). Esta story dá **visibilidade** a esse processo e permite ao usuário **aceitar** (ou ignorar) a sugestão de refino de segmento — a única parte do Scout que toca um campo declarado, e só com consentimento explícito (Decisão #2 do epic).

## Acceptance Criteria

1. **AC1:** Novo endpoint backend (ex.: `GET /onboarding/scout`) que retorna `scout_status` e, quando `done`, um resumo do `scout_insights` (incluindo `suggested_segment`, se houver).
2. **AC2:** No onboarding, enquanto `scout_status` for `running`/`pending`, exibir um indicador não-bloqueante: "Analisando seu perfil do Instagram para personalizar seu conteúdo…". O usuário **pode concluir o onboarding normalmente** sem esperar.
3. **AC3:** Quando o Scout conclui (`done`) e há `suggested_segment` divergente do declarado, exibir um card dispensável: mostra o nicho detectado e oferece "Usar este segmento" (aceitar) ou "Manter o meu" (ignorar).
4. **AC4:** Aceitar a sugestão chama um endpoint (ex.: `POST /onboarding/scout/accept`) que **atualiza o `segment` declarado** no `brand_profile` — esta é a única via pela qual o Scout altera um campo declarado, e sempre por ação do usuário.
5. **AC5:** `scout_status` = `skipped`/`failed` (conta vazia, privada, erro) → **nada é exibido** sobre o Scout; o onboarding segue idêntico ao fluxo sem Scout. Sem mensagem de erro ao usuário.
6. **AC6:** Nenhuma etapa desta story bloqueia a conclusão do onboarding — o Scout é sempre um enriquecimento opcional.

### Completion Notes List
- **LIMITAÇÃO HONESTA — teste manual não realizado:** a story exige `manual-test` no `quality_gate_tools`, e o AC do fluxo completo (conectar → ver status → aceitar/ignorar sugestão) exigiria um usuário autenticado real e uma conexão Instagram real. Não tenho ferramenta de browser leve nem credenciais de teste disponíveis nesta sessão — perguntei ao usuário antes de prosseguir e ele optou por validar visualmente depois, por conta própria. **Não fingi ter testado o que não testei.** Compensei com: typecheck limpo, verificação de servidor via curl (sem erro 500), e uma revisão de código linha a linha do componente que encontrou e corrigiu um bug real antes mesmo do @qa (ver próximo item).
- **Bug encontrado na autorevisão (antes do @qa):** `scout_status` tem `server_default='pending'` no banco (Story 22.3) para TODO cliente, inclusive quem nunca conectou o Instagram (fluxo "Conectar depois" em `StepConnectInstagram`). A implementação original mostraria "Analisando seu perfil…" por até 60s para esse usuário, mesmo que nenhuma análise tenha sido disparada — enganoso. Corrigido: o indicador visual só aparece quando `scout_status === "running"`; o estado `"pending"` continua fazendo polling em silêncio (sem custo visível), permitindo transição rápida para `"running"` quando a task realmente inicia, sem nunca mostrar nada para quem pulou a conexão.
- **Segurança no `POST /onboarding/scout/accept`:** o endpoint **ignora** qualquer valor de segmento enviado pelo cliente — lê `suggested_segment` diretamente do `brand_profile` já persistido no banco (só quando `scout_status == "done"`). Testado explicitamente (`test_accept_scout_suggestion_ignores_client_submitted_segment`) — impede que um cliente malicioso ou um bug no frontend grave um segmento arbitrário fora do que o próprio Scout já validou contra `FIXED_SEGMENTS`.
- **`GET /onboarding/scout` nunca vaza `scout_insights` de execuções não concluídas:** mesmo que `brand_profile["scout_insights"]` tenha dado de uma execução anterior, só é retornado quando `scout_status == "done"` — testado (`test_get_scout_status_skipped_no_insights`).
- **Polling client-side:** 5s de intervalo, até 12 tentativas (~60s), desiste silenciosamente depois disso (AC6 — nunca bloqueia). Cleanup correto via flag `cancelled` + `clearTimeout` no retorno do `useEffect` — evita atualização de estado após desmontagem do componente (ex.: usuário navega para o dashboard antes do polling terminar).
- **Local escolhido para a UI:** `StepComplete.tsx`, não `StepConnectInstagram.tsx` — é o step que o usuário efetivamente atinge após `connected=true` (o fluxo pula direto para lá), decisão já justificada no pre-flight.
- **AC5 (skipped/failed → nada exibido):** já é o comportamento padrão — `scoutSuggestion` só é setado quando `status === "done" && suggested_segment` presente; nenhum branch de erro é renderizado.
