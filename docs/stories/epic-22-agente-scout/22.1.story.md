## Story
**As a** sistema Scout,
**I want** buscar os posts recentes do Instagram do cliente conectado,
**so that** eu tenha matéria-prima real (legendas + imagens + métricas básicas) para analisar o perfil dele.

## Contexto / Motivação

Esta é a **fundação do Epic 22** e a story que resolve a **Decisão #1** do epic: validar contra a API real exatamente quais campos o escopo `instagram_business_basic` (já concedido no OAuth) devolve para `/me/media`.

O token e o `instagram_business_id` já estão persistidos no `Client` após a conexão OAuth (`app/api/meta.py:meta_callback`). O que falta é uma tool que use esse token para ler a mídia própria do cliente. Nenhum novo escopo é necessário — `instagram_business_basic` cobre leitura de mídia própria.

## Acceptance Criteria

1. **AC1:** Nova tool `app/tools/instagram_media.py` com função `fetch_recent_media(ig_id: str, access_token: str, limit: int = 12) -> list[dict]` que chama `graph.instagram.com/me/media` (ou `{ig_id}/media`) e retorna uma lista de posts.
2. **AC2:** Cada item retornado contém: `id`, `caption`, `media_type`, `media_url`, `thumbnail_url`, `permalink`, `timestamp`, `like_count`, `comments_count`. **Confirmado via documentação oficial da Meta (Instagram Platform / Instagram Login, `graph.instagram.com`)** que todos os 9 campos estão disponíveis sob o escopo `instagram_business_basic` já concedido — ver Dev Notes para a referência. `thumbnail_url` é incluído por decisão de pre-flight (Story 22.2 precisará dele para posts de vídeo, evitando extração de frames).
3. **AC3:** Conta sem posts retorna `[]` (lista vazia) — não é erro.
4. **AC4:** Falha de rede, token inválido (erro #190) ou conta privada retornam `[]` e logam o motivo via `logger.warning` — **a tool nunca levanta exceção para o chamador** (degradação graciosa; o Scout depende disso).
5. **AC5:** A tool respeita o `limit` (padrão 12) para conter custo/latência da análise a jusante.
6. **AC6 (adicionado — pre-flight):** Nova função `fetch_account_info(access_token: str) -> dict | None` na mesma tool, buscando dados de conta via `graph.instagram.com/me` (`fields=username,biography,followers_count,media_count,profile_picture_url`). Resolve por completo a Decisão #1 do epic (bio do perfil é acessível — confirmado via documentação oficial). Retorna `None` em qualquer falha (mesmo padrão defensivo do AC4) — nunca levanta exceção.
7. **AC7:** Testes unitários com mock via `respx` cobrindo: caminho feliz (N posts) de `fetch_recent_media`, conta vazia (`[]`), erro #190 (`[]` + log), resposta malformada (`[]` + log), `fetch_account_info` caminho feliz, e `fetch_account_info` falha (`None`).

## Dev Notes

- Padrão de chamada Graph API já existe em `app/agents/instagram_analytics.py` (`_meta_get` com tratamento do erro #190) — **reusar o padrão**, mas atenção: analytics usa `graph.facebook.com/v21.0`; a leitura de mídia sob Instagram Login deve usar `graph.instagram.com` (mesma base de `app/core/meta_oauth.py:INSTAGRAM_GRAPH_BASE`).
- O escopo concedido está em `app/core/meta_oauth.py:15` → `instagram_business_basic,instagram_business_content_publish`.
- Não persistir nada nesta story — apenas ler e retornar. Persistência é da Story 22.3.
- **Validação real da API é parte do trabalho:** se algum campo do AC2 não vier sob o escopo atual, documentar e ajustar o contrato do `ScoutReport` (Story 22.2) de acordo — não assumir.

### Referência confirmada (pre-flight, 2026-07-22)

Sem token real conectado neste ambiente de dev, a validação foi feita contra a documentação oficial da Meta (Instagram Platform — Instagram Login, não o fluxo antigo via Facebook Login):

**IG Media** (`developers.facebook.com/docs/instagram-platform/reference/instagram-media/`) — todos sob `instagram_business_basic`: `id`, `caption`, `media_type`, `media_url`, `thumbnail_url`, `permalink`, `timestamp`, `like_count`, `comments_count`.

**IG User** (`developers.facebook.com/docs/instagram-platform/instagram-graph-api/reference/ig-user/`) — todos sob `instagram_business_basic`: `username`, `biography`, `followers_count`, `media_count`, `profile_picture_url`.

Nenhum campo exige escopo adicional. **Recomendação:** validar com um token real de conta conectada assim que possível (ex.: após a Story 22.3 estar ligada ao fluxo real) — a doc pode divergir de comportamento real de edge cases (ex.: conta sem bio preenchida).

**Endpoint:** usar `{ig_id}/media` e `{ig_id}?fields=...` (não `/me/media`) para manter consistência com `app/agents/instagram_analytics.py` (`f"{ig_business_id}/insights"`), que já é o padrão do codebase para chamadas por-conta na Graph API.

### Refactoring Performed

Durante a análise de rastreabilidade (AC2/AC5/AC6), identifiquei que os testes verificavam apenas o *retorno* das funções, nunca os *parâmetros efetivamente enviados* na requisição HTTP (`fields=...`, `limit=...`). Isso é uma lacuna real: os ACs afirmam que campos e limite específicos são solicitados, mas nenhum teste provava isso — um refactor futuro que corrompesse a string `MEDIA_FIELDS` ou esquecesse de passar `limit` passaria pela suíte sem ser notado.

- **File**: `backend/tests/test_instagram_media.py`
  - **Change**: Adicionados `test_fetch_recent_media_requests_correct_fields_and_limit` e `test_fetch_account_info_requests_correct_fields`, inspecionando `route.calls.last.request.url.params` via respx.
  - **Why**: AC2, AC5 e AC6 fazem afirmações específicas sobre quais parâmetros são enviados à API — sem essa verificação, o contrato central desta story (resolver a Decisão #1 do epic) não era protegido por teste.
  - **How**: Fecha a lacuna sem alterar o código de produção. Suíte completa re-executada após a mudança: 312/312 passando (310 anteriores + 2 novos), zero regressão.
