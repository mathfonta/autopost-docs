## Story
**As a** sistema,
**I want** persistir o ScoutReport, mesclá-lo de forma aditiva no `brand_profile` e disparar a análise de forma assíncrona logo após a conexão OAuth,
**so that** a próxima geração de conteúdo use o perfil enriquecido sem que o onboarding jamais espere pela análise.

## Contexto / Motivação

Story que conecta o Scout ao ciclo de vida real do cliente e resolve a **Decisão #2** do epic (auto-aplicar vs sugerir). O `brand_profile` (JSONB no `Client`) é a única superfície de calibração que os agentes a jusante já lêem (`app/tasks/pipeline.py:54` carrega `client.brand_profile` e passa aos agentes). O Scout enriquece esse mesmo campo — nada de mensageria entre agentes.

O disparo é **assíncrono via Celery**: o onboarding termina na hora, o Scout roda em background.

## Acceptance Criteria

1. **AC1:** Migração Alembic adiciona ao `Client`: `scout_report` (JSONB, nullable — relatório cru para auditoria) e `scout_status` (String, default `pending`: `pending | running | done | skipped | failed`).
2. **AC2:** Nova tarefa Celery `run_scout_analysis(client_id)` (em `app/tasks/`) que: carrega o Client, chama `fetch_recent_media` (22.1) → `analyze_profile` (22.2), persiste `scout_report`, atualiza `scout_status` e faz o merge no `brand_profile`.
3. **AC3:** O merge é **aditivo**: o ScoutReport entra em `brand_profile["scout_insights"]`. Os campos declarados pelo usuário (`segment`, `tone`, `company_name`, `city`) **nunca são sobrescritos** por esta task.
4. **AC4:** Divergência de segmento (`suggested_segment` do ScoutReport) é gravada dentro de `scout_insights` como **sugestão** — nunca aplicada automaticamente ao `segment` declarado (a aplicação, se o usuário aceitar, é da Story 22.5).
5. **AC5:** `meta_callback` (`app/api/meta.py`), no ramo de **sucesso** (após `db.commit()`), enfileira `run_scout_analysis(client_id)` via `.delay()`. A falha ao enfileirar não pode quebrar o callback de sucesso — o redirect `connected=true` acontece de qualquer forma.
6. **AC6:** `analyze_profile` retornando `None` (mídia vazia ou falha, da 22.2) → `scout_status = skipped` (ou `failed`), `brand_profile` intocado, nenhuma exceção propagada.
7. **AC7:** Transições de status corretas: `pending` → `running` no início da task → `done | skipped | failed` no fim.
8. **AC8:** Testes: migração aplica/reverte; task no caminho feliz (merge aditivo, status `done`); mídia vazia (`skipped`, brand_profile intocado); disparo enfileirado no sucesso do callback (sem regressão na Story 12.4); merge preserva campos declarados.

## Dev Notes

- **Precedente de task por-cliente com token Meta:** `app/tasks/pipeline.py:757` (`renew_meta_tokens`) — mesmo padrão de carregar Client, usar `meta_access_token`, commitar. Usar como molde.
- `celery_app` está em `app/tasks/__init__.py`; tasks são `@celery_app.task(bind=True, name="...")`; enfileira com `.delay()` / `.apply_async()`.
- Ponto de disparo exato: `app/api/meta.py:127-133` (ramo de sucesso do `meta_callback`, logo após `await db.commit()`).
- **Não modificar** os campos declarados do brand_profile é regra dura (AC3) — o valor do epic depende de o usuário confiar que o que ele declarou é respeitado.
- Migrações seguem o padrão do diretório de migrations existente (verificar `alembic/versions` ou equivalente antes de criar).
